Entry Point

main.go

The single binary entry point — nothing more.

Entry main.go

Contains only the main() function. Passes os.Args[1:] to analyzer.Execute() and exits with the integer return code. All real logic lives elsewhere — this file is intentionally trivial so the CLI dispatcher can be tested independently of the OS process lifecycle.

Declares package main — makes this the binary entry point.
Hands off all arguments to analyzer.Execute() and propagates its exit code.
Intentionally contains zero business logic — the entire program can be exercised by calling Execute() directly in tests.
Calls → cmd/analyzer.Execute()

Command Layer

cmd/analyzer/

Six files that collectively implement the CLI: routing, four user-facing commands, and the shared analysis pipeline that all commands share.

Dispatcher root.go

The CLI router. Execute() reads the first argument and dispatches to the correct command handler. It also initialises the global logger before any command runs, honouring --verbose and --debug flags found anywhere in the argument list.

Exposes Execute(args []string) int — the public entry point called by main.go.
Holds GlobalConfig (verbose flag + logger) shared by all subcommands.
Routes: analyze → runAnalyze, fetch → runFetch, stats → runStats, filter → runFilter.
Handles help and version as built-in sub-commands, printing the full usage text.
Returns exit code 1 for unknown commands, 0 for help/version.
Called by → main.go
Calls → runAnalyze · runFetch · runStats · runFilter internal/utils.NewLogger
Command analyze.go

Implements the analyze command — the most commonly used entry point when working with a CSV export from GCP Security Command Center. Also owns the shared applyFilters() and resolveFormats() helpers used by the pipeline, as well as the terminal summary printer.

Defines AnalyzeFlags struct — holds every CLI flag value for the analyze/fetch commands.
Parses all CLI flags via flag.FlagSet after separating the positional CSV path from named flags using utils.ExtractPositional.
Calls parser.ParseFile() to load CSV data, then hands findings to runPipeline().
applyFilters() — filters by priority, category, project, and risk-score range. Used by both analyze and filter commands.
resolveFormats() — picks output formats: --formats > --format > default (markdown + html).
printSummary() — prints the terminal dashboard (total/critical/high/medium/low counts, mean risk score, range).
Calls → pkg/parser.ParseFile pipeline.runPipeline internal/utils.ExtractPositional
Command fetch.go

Implements the fetch command — the live API variant of analyze. Connects to GCP Security Command Center in real time, retrieves all active findings within a configurable lookback window, then routes them through the identical analysis pipeline as the analyze command.

Defines FetchFlags which embeds AnalyzeFlags, adding --org-id, --days, and --save-csv.
Creates a cancellable context.Context hooked to os.Interrupt so Ctrl+C cleanly aborts a long fetch.
Delegates to fetcher.NewFetcher().FetchFindings() then passes results to runPipeline() with SaveCSVPath set when --save-csv is given.
Exits 0 with a "no findings" message if the SCC API returns an empty result set — not an error.
Calls → pkg/fetcher.FetchFindings pipeline.runPipeline
Command stats.go

Implements the stats command — a read-only, no-output-file mode that parses a CSV, scores it, detects compliance violations, then prints a structured dashboard to stdout. Nothing is written to disk.

Runs the full parse + score + compliance chain (same as the pipeline) but skips remediation, AI enrichment, filtering, and file writing.
Prints: total rows / parse errors / active findings; priority breakdown with percentages; risk score stats (mean, median, stddev, range); top 10 categories; compliance violations by framework; top 5 GCP projects by finding count.
Useful as a fast situational overview before committing to a full report run on large datasets.
Calls → pkg/parser.ParseFile pkg/scoring.Engine.ScoreAll pkg/compliance.Detector pkg/report.Builder.Build
Command filter.go

Implements the filter command — a CSV-to-CSV slicing tool. Reads an input CSV, scores it so risk-score filters work, applies the requested filters, then writes just the matching rows to a new CSV (or stdout). No HTML/Markdown report is generated.

Supports --priority, --category, --project, --min-risk-score, --max-risk-score.
When -o is omitted the filtered CSV is written to stdout, making it pipeable to other tools.
Calls applyFilters() defined in analyze.go — the same filter logic, zero duplication.
Calls → pkg/parser.ParseFile pkg/scoring.Engine.ScoreAll analyze.applyFilters pkg/report.CSVGenerator
Shared Logic pipeline.go

The heart of the command layer. Defines runPipeline() — the single function that both the analyze and fetch commands call after their input-gathering phase. Executes the full seven-stage chain in strict order, guaranteeing the same output quality regardless of whether data came from a CSV file or the live API.

Stage 1 — Score: calls scoring.Engine.ScoreAll() — assigns a 0–100 risk score to every finding in parallel.
Stage 2 — Compliance: calls compliance.Detector.DetectViolations() per finding.
Stage 3 — Remediation (optional): calls remediation.Generator.GenerateAll() when --include-remediation is set (default on).
Stage 4 — AI Enrichment (optional): calls llm.Enricher.EnrichAll() when --ai-enhance is set and ANTHROPIC_API_KEY is present.
Stage 5 — Save raw CSV (optional): writes the scored-but-unfiltered findings as CSV when --save-csv is given.
Stage 6 — Filter: calls applyFilters() to reduce findings to the requested subset.
Stage 7 — Report: builds the Report object, resolves output paths, calls report.WriteReports() or WriteSplitByPriority(), then writes per-finding remediation scripts.
Called by → analyze.go · fetch.go
Calls → pkg/scoring pkg/compliance pkg/remediation pkg/llm pkg/report

Package — fetcher

pkg/fetcher/

Connects to the GCP Security Command Center gRPC API and converts raw protobuf responses into the internal Finding model.

Package fetcher.go

The GCP API client. Authenticates using Application Default Credentials, builds a time-filtered query for ACTIVE findings, and paginates through all results (up to 1 000 per page) automatically.

Creates a securitycenter.Client — authentication happens here via Google's ADC chain (env var, service account key, or user credentials).
Builds the filter string: state="ACTIVE" AND event_time >= "{cutoff}" using the requested number of lookback days.
Target is organizations/{orgID}/sources/- — the wildcard - means all security sources across the entire organization.
Loops over the iterator until iterator.Done, calling convert.ConvertFinding() on each result. Logs progress every 500 findings.
Returns a partial result + error on mid-fetch failure rather than discarding all data already retrieved.
Called by → cmd/analyzer/fetch.go
Calls → fetcher/convert.ConvertFinding GCP securitycenter API (gRPC)
Package convert.go

Translates a raw GCP protobuf ListFindingsResponse_ListFindingsResult into the internal *models.Finding. This is the boundary between the GCP API world and the rest of the program.

Extracts core fields: name, category, state, severity, resource name/type/project, timestamps, description, next steps, external URI.
Extracts vulnerability data: CVE ID, CVSS base score, zero-day flag, exploit-in-the-wild flag, exploitation activity level.
Extracts compliance mappings from the pb.GetCompliances() list and serialises them as JSON for the compliance detector.
sanitizeEnum() strips GCP protobuf prefix noise (e.g. FINDING_CLASS_VULNERABILITY → VULNERABILITY).
sanitizeExploitActivity() maps GCP's enum values (WIDE, AVAILABLE, etc.) to the scoring engine's vocabulary (ACTIVE, POC, etc.).
extractProjectID() parses "projects/my-project" paths to return just the project ID string.
Called by → fetcher.go
Produces → internal/models.Finding

Package — parser

pkg/parser/

Reads GCP Security Command Center CSV exports and converts each row into an internal Finding struct, handling the inconsistent column naming and embedded JSON that GCP produces.

Package csv.go

The CSV parser. Handles GCP's varying column naming conventions (both bare names like severity and full API paths like finding.severity), embedded JSON sub-objects (CVSS data, compliance arrays), and malformed rows — without crashing.

columnAliases — a static map of ~30 GCP SCC column name variants to canonical internal names, enabling the parser to handle multiple GCP export formats transparently.
buildIndex() — two-pass header scan: first indexes every column exactly, then overlays the alias map. "First column wins" prevents duplicates.
ParseFile() / Parse() — opens the file and streams rows. Bad rows increment a parseErrors counter and are skipped rather than aborting the entire parse.
parseRow() — maps a single CSV row to a Finding. Applies defaults (severity → LOW, state → ACTIVE) for missing fields.
parseCVSSScore() — extracts the numeric base score from a JSON object, a JSON string, or a plain float — whichever format GCP happened to emit.
parseComplianceFrameworks() — handles compliance data as a JSON array of objects, a JSON array of strings, or a plain comma-separated string.
Uses csv.Reader with LazyQuotes=true and variable field count to tolerate GCP's occasionally malformed output.
Called by → analyze.go · stats.go · filter.go
Produces → internal/models.Finding

Package — scoring

pkg/scoring/

Two files: one computes each finding's composite 0–100 risk score using six weighted components; the other provides filtering, sorting, grouping, and statistics utilities.

Package risk.go

The risk scoring engine. Calculates a composite 0–100 score per finding from six independent components, applies a category weight multiplier, enforces a GCP-severity floor, and writes a human-readable rationale. Uses up to 8 goroutines to score all findings in parallel.

ScoringConfig — holds all weight constants. DefaultConfig() returns production values; can be overridden from a YAML file.
baseSeverityScore() — maps GCP severity label to a base score: Critical=40, High=30, Medium=20, Low=10.
cvssScore() — scales the CVSS base score (0–10) by 3× to a 0–30 component, capped at 30.
exploitabilityScore() — adds up to 20 points for: exploited in the wild (+10), zero-day (+8), active exploit code (+6), known CVE (+2).
classModifierScore() — adds up to 10 points based on finding class: THREAT=10, VULNERABILITY=7, MISCONFIGURATION=5, OBSERVATION=2.
exposureScore() — adds up to 10 points for public IP (+5), load-balancer category (+3), data store category (+2).
complianceScore() — adds up to 10 points when compliance frameworks are mapped or audit/logging categories are present.
categoryWeight() — multiplies the sum by 1.2× for high-risk categories (vulnerabilities, privilege escalation), 1.0× for medium, 0.8× for low-risk.
Severity floor: CRITICAL findings score minimum 75, HIGH minimum 55 — GCP's expert assessment is never downgraded by missing metadata.
ScoreAll() distributes findings across up to 8 goroutines via a buffered job channel, blocking until all workers finish.
Called by → pipeline.go · stats.go · filter.go
Writes to → models.Finding.RiskScore · models.Finding.Priority
Package priorities.go

Utility functions that operate on scored findings: filtering by priority/category/project/score, sorting by descending risk score, grouping into priority buckets, and computing statistical summaries (mean, median, stddev, range) over a score distribution.

FilterByPriority / Category / Project / RiskScore() — each returns a filtered sub-slice, all using set-based membership for O(n) performance.
SortByRiskScore() — descending sort; nil RiskScore fields are treated as 0.0.
GroupByPriority() — returns a map with pre-populated keys for all four priority levels, so callers can safely index any key even if empty.
ComputeStats() — collects scores from all findings, sorts them, and returns a RiskScoreStats with count, mean, median, min, max, and stddev.
Called by → pkg/report/builder.go

Package — compliance

pkg/compliance/

Maps findings to the compliance frameworks they violate. Two files: one defines the canonical list of supported frameworks; the other detects and aggregates violations from finding data.

Package frameworks.go

Defines the seven supported compliance frameworks and their alias normalization. Acts as the single source of truth for framework naming — all other code calls into this file to resolve a raw name to a canonical ID.

KnownFrameworks — the master list: CIS Benchmarks, PCI-DSS, HIPAA, SOC 2, ISO 27001, NIST, GDPR. Each entry has an ID, display name, description, and list of known aliases.
aliasMap — built at init time, maps every lowercase-normalized alias string to its canonical ID ("cisgcp" → "CIS", "pci_dss" → "PCI-DSS", etc.).
normalize() — strips hyphens, underscores, spaces, and lowercases — so "CIS_GCP", "CISGCP", and "cis-gcp" all resolve to the same key.
CanonicalID() — public function used by the detector to resolve any observed string to a canonical ID, returning the input unchanged if no match.
Called by → compliance/detector.go
Package detector.go

Parses compliance data embedded in each finding (from two different JSON fields GCP uses), deduplicates violations, and aggregates them into a framework → violations map for reporting.

DetectViolations(f) — runs both parsers, deduplicates with dedupViolations(), and attaches the result to f.Violations.
parseCompliances() — parses the finding.compliances JSON array. Handles both array and single-object forms, and alternate field names (standard / framework / name).
parseComplianceDetails() — parses the finding.compliance_details.frameworks JSON field (array or object form).
Aggregate(findings) — collapses all per-finding violations into a map[string][]*ComplianceViolation keyed by framework, sorted by violation count descending. Used to build the compliance summary section in reports.
dedupViolations() — removes duplicate framework+control combinations using a seen-set, preserving order.
Called by → pipeline.go · stats.go · filter.go
Calls → frameworks.CanonicalID

Package — remediation

pkg/remediation/

Generates actionable fix guidance for every finding and complete, runnable bash/python scripts for CRITICAL findings with real resource names baked in.

Package guidance.go

Generates structured human-readable remediation guidance for every finding: a one-line summary, step-by-step instructions, effort estimate, automation potential rating, documentation links, and priority rationale. For CRITICAL findings it also triggers script generation.

Generate(f) — assembles a complete RemediationStep and attaches it to f.Remediation. For CRITICAL findings calls ScriptGenerator.Generate(f).
parseNextSteps() — converts the raw next_steps string from GCP (which can be a numbered list, bullet list, or semicolon-separated) into a clean string slice.
resourceLinks() — returns category-specific GCP documentation URLs (e.g. VPC flow logs docs for FLOW_LOGS, firewall docs for FIREWALL). Always appends the generic SCC overview link. Appends NVD link when a CVE ID is present.
estimatedEffort() — returns a plain-English effort estimate (Low / Medium / High) keyed to the finding category and class.
automationPotential() — returns an automation level and a sample gcloud command demonstrating how to automate the fix.
summary() — one-line description of what needs to change, category-aware (e.g. "Enable VPC Flow Logs on the affected subnet").
priorityRationale() — explains why the finding got its priority: lists the contributing factors (severity, CVSS, exploit status, public IP, compliance).
Called by → pipeline.go
Calls → remediation/scripts.ScriptGenerator
Package scripts.go

Generates complete, immediately runnable remediation scripts for CRITICAL findings. Scripts include the real resource names, project IDs, regions, and zones extracted from the GCP resource path — so they can be executed directly rather than requiring manual substitution. Supports both bash and Python 3.

Generate(f) — dispatches to the correct script generator based on finding category. Unrecognised categories get a generic inspection script rather than failing.
parseResourceName() — parses GCP API resource paths like //compute.googleapis.com/projects/proj/regions/us-central1/subnetworks/my-subnet into a named key-value map.
scriptHeader() — emits a standardised comment block at the top of every script with finding name, category, priority, risk score, resource, project, CVE (if any), and generation date.
All bash scripts include set -euo pipefail and a DRY_RUN environment variable guard — set it to true to preview without making changes.
Bash scripts cover: VPC flow logs, Private Google Access, GCS bucket access logging, audit logging (with inline Python policy merging), firewall rule logging, open firewall restrictions, weak SSL policy creation, service account key rotation, IAM anomaly investigation.
Python 3 scripts cover: container image vulnerability remediation (queries Container Analysis API, gets cluster credentials, prints rebuild instructions) and OS vulnerability patching (queries OS Config API, launches a Cloud OS Config patch job).
Called by → remediation/guidance.go

Package — llm

pkg/llm/

Optional AI enrichment layer. Calls the Claude API to improve the risk rationale and remediation script for CRITICAL findings. Completely skipped if --ai-enhance is absent or ANTHROPIC_API_KEY is unset.

Package enricher.go

An HTTP client for the Claude Messages API. Sends a structured prompt for each CRITICAL finding, parses the JSON response, and overwrites the finding's Rationale and RemediationScript with the AI-improved versions. Runs up to 4 enrichments concurrently.

Uses claude-haiku-4-5 model with 1 024 max tokens and a 30-second HTTP timeout — balanced for cost and speed at scale.
EnrichAll() — filters to CRITICAL findings, distributes them across up to 4 goroutines via a job channel. Per-finding errors are logged and skipped; the original template values are preserved.
buildPrompt() — constructs a structured prompt with all available finding context: category, severity, resource name, project, description, CVE, CVSS. Instructs Claude to return only a JSON object with three fields.
parseResult() — strips accidental markdown fences from the response then unmarshals the JSON into enrichResult{RiskRationale, RemediationScript, RemediationLang}.
Only overwrites finding fields if the LLM returned non-empty strings — partial failures degrade gracefully to template output.
Appends [AI-enhanced] to the rationale string so downstream consumers can distinguish AI output from template output.
Called by → pipeline.go
Calls → api.anthropic.com/v1/messages (HTTP)

Package — report

pkg/report/

Five files that turn scored findings into output. One orchestrates everything (builder.go); four render findings into specific formats (HTML, Markdown, JSON, CSV).

Package builder.go

The report orchestrator. Assembles the *models.Report object from findings, computes all aggregate statistics, and exposes functions for writing reports in one or more formats, optionally split by priority level, and for writing per-finding remediation scripts.

Build() — creates the Report object: sets metadata (timestamp, input file, total rows, parse errors), computes stats, category breakdown, and project breakdown.
computeStats() — counts findings by priority, computes risk score statistics (mean, median, stddev via utils), builds top-categories and top-projects ranked lists.
buildCategoryBreakdown() / buildProjectBreakdown() — aggregates per-category and per-project counts with priority sub-breakdown and average risk score.
WriteReport() — dispatches to the correct renderer (Markdown / JSON / HTML / CSV) and writes to a file path.
WriteReports() — writes the same report in multiple formats into an output directory, creating it if needed.
WriteSplitByPriority() — writes separate files per priority level (critical, high, medium, low), skipping levels with zero findings.
WriteRemediationScripts() — creates remediation-scripts/ subdirectory, writes one .sh or .py file per CRITICAL finding that has a script, sets chmod 755 on each.
Called by → pipeline.go · stats.go · filter.go
Calls → html.go · markdown.go · json.go · csv.go
Package html.go

Renders the Report as a self-contained HTML file with styling embedded inline. Produces the richest output format — includes executive summary charts, per-finding cards with all metadata, compliance tables, and collapsible remediation sections.

Implements the HTMLGenerator struct. Called by builder.WriteReport() when format is "html".
Outputs a complete single-file HTML page — no external CSS or JS dependencies.
Priority badges, risk score bars, compliance framework tables, and remediation steps are all rendered from the structured models.Report data.
Called by → builder.WriteReport
Package markdown.go

Renders the Report as a GitHub-flavoured Markdown document. Optimised for reading in terminals, git forges, and documentation sites. Tables, headers, code blocks, and bullet lists make findings easy to scan without a browser.

Implements MarkdownGenerator. Default output format when no explicit format is requested.
Renders an executive summary table, then findings grouped and sorted by risk score, with remediation steps as numbered lists and scripts in fenced code blocks.
Called by → builder.WriteReport
Package json.go

Renders the Report as pretty-printed JSON. Intended for integration with SIEMs, dashboards, custom scripts, and any downstream tooling that needs machine-readable structured output.

Implements JSONGenerator. The entire models.Report struct serialises directly since all fields carry json: tags.
Supports a pretty-print flag (enabled by default via NewJSONGenerator(true)).
Called by → builder.WriteReport
Package csv.go

Renders findings as a CSV file compatible with spreadsheets and data analysis tools. Each finding becomes one row; scored and compliance-annotated fields are included alongside the original GCP fields.

Implements CSVGenerator. Used by both the report pipeline and the filter command's direct CSV output.
Adds computed columns (risk score total, priority, rationale) not present in the original GCP export, making the output self-contained for re-analysis.
Called by → builder.WriteReport · filter.go

Internal — models

internal/models/

The shared data layer. Three files that define every struct, constant, and type used across all packages. No business logic lives here — only data shapes.

Model finding.go

Defines the Finding struct — the central data object that flows through every stage of the pipeline. Contains all raw fields from GCP (CSV tags), all parsed/computed fields added during analysis (csv:"-"), and pointer fields for attached enrichments.

Core identification: name, finding class, finding type, category, state, severity.
Resource fields: resource name, display name, type, project ID, project display name.
Timestamps: EventTime (parsed) and EventTimeRaw (string) — dual fields for both rendering and API use.
Vulnerability data: CVE ID, CVSS score, exploit-in-the-wild flag, zero-day flag, exploitation activity, public IP exposure — parsed from raw JSON strings.
Computed fields (added during pipeline): *RiskScore, Priority, *RemediationStep, []ComplianceViolation.
Helper methods: IsActive(), HasCVE(), SeverityScore() (numeric for sorting), ShortName() (last path segment of the finding name).
Model report.go

Defines the Report struct and all supporting aggregate types. This is the data contract between the report builder and the format renderers.

Report — top-level container: metadata (timestamp, input file, row counts), slice of Findings, aggregate Stats, ComplianceSummary, CategoryBreakdown, ProjectBreakdown.
ReportStats — total/critical/high/medium/low counts, risk score statistics, top-categories list, top-projects list.
CategoryStats / ProjectStats — per-category and per-project breakdowns with priority sub-counts and average risk scores.
FilterOptions / ReportOptions — option structs used as a clean API boundary between commands and the pipeline.
Model risk_score.go

Defines all scoring-related and remediation-related types, the four priority constants, the numeric thresholds that map scores to priority levels, the SLA strings, and the PriorityFromScore() function.

RiskScore — holds all six component values, the category weight, the total, and the human-readable rationale string.
RiskScoreStats — statistical summary: count, mean, median, min, max, stddev.
ComplianceViolation — a single framework+control violation: framework name, version, control ID, description, finding names, and occurrence count.
RemediationStep — the full remediation record: summary, next steps list, documentation links, effort estimate, automation level + hint, priority rationale, script content, and script language.
Priority constants + thresholds: Critical ≥ 75, High ≥ 55, Medium ≥ 35, Low < 35. SLAs: Critical = 24–48 h, High = 1 week, Medium = 30 days, Low = 90 days.
PriorityFromScore() — converts a numeric score to the corresponding priority string constant.

Internal — utils

internal/utils/

Three files providing shared foundations: a structured logger, math and string helper functions, and a CLI argument parsing utility.

Utility logger.go

A minimal structured logger with four levels (DEBUG, INFO, WARN, ERROR). Writes to stderr with ISO-8601 timestamps so it does not pollute stdout (which is reserved for report content and the terminal summary).

Logger struct — holds a minimum level and an io.Writer. Log calls below the minimum level are silently dropped.
Format: 2006-01-02T15:04:05Z [INFO ] message — fixed-width level strings align log output for readability.
DefaultLogger — package-level singleton at INFO level, used by packages that haven't received an injected logger.
SetLevel() — runtime level adjustment; called by commands honouring --verbose and --debug flags.
Utility helpers.go

A collection of pure, stateless helper functions used across multiple packages: floating-point math, string manipulation, and safe numeric conversions. Contains no state.

Math: Min / Max / Round / Mean / Median / StdDev — used by the scoring engine and report builder for risk score statistics.
ContainsAny(s, substrings...) — case-insensitive multi-substring check used extensively by the scoring and remediation packages for category matching.
Truncate(s, maxLen) — safe string truncation with "..." suffix for display.
NormalizeString() — normalises line endings and trims whitespace.
ParseBool() — maps "true"/"yes"/"1"/"t"/"y" → true, everything else → false. Used by the CSV parser for boolean GCP fields.
SafePercentage(part, total) — division with zero-guard; used in the stats command's percentage column.
UniqueStrings() — order-preserving deduplication of string slices; used by the CSV parser for compliance framework lists.
StringSliceContains() — case-insensitive membership test.
Utility flags.go

Solves a specific problem with Go's standard flag package: it requires positional arguments to come last. GCP-Sec allows users to place the CSV filename anywhere in the argument list — this file provides the parsing logic that makes that work.

ExtractPositional(args, knownValueFlags) — single-pass scan of the argument list. Recognises named flags (starting with -) and the values that follow them, so it never mistakes a flag's value for the positional CSV path.
Returns (positional string, flagArgs []string) — the CSV path and a cleaned args slice that Go's FlagSet can safely parse.
Handles both --flag value and --flag=value syntax via the knownValueFlags map and a skipNext boolean.
Called by analyze.go, stats.go, and filter.go before their FlagSet parse calls.
Called by → analyze.go · stats.go · filter.go
