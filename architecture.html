GCP Security Analyzer

Architecture & End-to-End Workflow Documentation

Overview

GCP Security Analyzer is a command-line tool that ingests security findings from Google Cloud Security Command Center (SCC) — either from a CSV export or directly via the live SCC API — and produces prioritized, enriched security reports in multiple formats (Markdown, HTML, JSON, CSV) along with executable remediation scripts.

Input (CSV export OR live GCP SCC API) │ ▼ Parse Findings → []*Finding (raw, unscored) │ ▼ Risk Scoring → RiskScore.Total (0-100) + Priority assigned │ ▼ Compliance Detection → Violations annotated per finding │ ▼ Remediation Guidance → Steps, effort, scripts (--include-remediation) │ ▼ AI Enrichment → Claude API overwrites rationale + scripts (--ai-enhance) │ ▼ Filtering → Subset by priority, category, project, risk score │ ▼ Report Generation → markdown + html (default); json / csv optional │ └─→ remediation-scripts/ *.sh *.py

CLI Commands

analyze

Primary command — CSV input

Reads a CSV export from GCP SCC and runs the full analysis pipeline.

GCP-Sec analyze <input.csv> [options]

-o, --output <path>       Output file; stem used for multi-format naming
-f, --format <fmt>        Single format: markdown | html | json | csv
    --formats <list>      Comma-separated (e.g. markdown,html,json)
-d, --output-dir <dir>    Output directory
-p, --priority <list>     Filter: CRITICAL,HIGH,MEDIUM,LOW
-c, --category <list>     Filter by finding category
    --project <list>      Filter by GCP project ID or display name
    --min-risk-score <n>  Minimum risk score (0-100)
    --max-risk-score <n>  Maximum risk score (0-100)
    --include-remediation  Generate remediation steps and scripts
    --include-compliance   Include compliance framework details
    --ai-enhance           Enrich CRITICAL findings via Claude AI
    --split-by-priority    Write separate file per priority level
Default behavior: no format flags → generates both .md and .html using the input filename stem. 
GCP-Sec analyze findings.csv
# → findings-report.md + findings-report.html

GCP-Sec analyze findings.csv -o security-report.md
# → security-report.md + security-report.html
fetch

Live SCC API

Connects to the GCP Security Command Center API and retrieves live findings without needing a CSV export. Accepts all the same options as analyze.

GCP-Sec fetch --org-id <ORG_ID> [options]

    --org-id <id>   GCP organization ID (required)
    --days <n>      Lookback window in days (default 7)
    --save-csv <p>  Also save raw findings as CSV before analysis

Requires Application Default Credentials:

gcloud auth application-default login
stats

Console summary (no files written)

GCP-Sec stats <input.csv> [-v]

Prints priority distribution, risk score statistics (mean, median, range, stddev), top categories, top projects, and detected compliance frameworks to stdout. No output files are written.

filter

Export filtered CSV

GCP-Sec filter <input.csv> [filter options] -o filtered.csv

Parses, scores, and filters findings, then writes the results as a new CSV. Useful for narrowing down findings for follow-up workflows.

Stage 1 Ingestion

Path A: CSV (pkg/parser/csv.go)

The parser reads a CSV produced by gcloud scc findings list --format=csv or exported from the GCP Console.

Column normalization: GCP's CSV uses dotted API paths as headers. The parser maps them to short canonical names via a columnAliases map:

CSV column (GCP API path)Canonical name
finding.categorycategory
finding.severityseverity
resource.project_idproject_id
finding.vulnerability.cve.idparsed → Finding.CVEID
finding.compliancesparsed → Finding.ComplianceFrameworks
finding.external_exposure.public_ip_addressparsed → Finding.PublicIPAddress

Processing steps:

  1. Read header row → build column-index map with alias lookups
  2. For each data row: extract fields, parse JSON sub-objects, parse timestamps
  3. Return []*models.Finding plus count of non-fatal parse errors

Path B: Live API (pkg/fetcher/)

fetcher.go connects to the GCP SCC API using Application Default Credentials and lists all ACTIVE findings for the organization within the lookback window, handling pagination automatically (1,000 findings per page).

convert.go converts protobuf response objects to *models.Finding: maps GCP enum values to clean strings (FINDING_CLASS_VULNERABILITY → VULNERABILITY), converts protobuf timestamps, and serializes nested fields to the JSON format expected by the compliance detector.

Stage 2 Risk Scoring

Every finding receives a numeric risk score from 0 to 100. Scoring runs in parallel across 8 goroutines (pkg/scoring/risk.go).

Scoring formula

Total = (BaseSeverity + CVSS + Exploitability + ClassModifier + Exposure + Compliance) × CategoryWeight
ComponentMax ptsHow it's calculated
Base Severity40CRITICAL=40, HIGH=30, MEDIUM=20, LOW=10
CVSS30CVSS score × 3, capped at 30
Exploitability20In-the-wild=+10, zero-day=+8, active=+6, POC=+4, low=+2, has CVE=+2
Class Modifier10THREAT=10, VULNERABILITY=7, MISCONFIGURATION=5, OBSERVATION=2
Exposure10Public IP=+5, internet-facing=+3, critical resource type=+2
Compliance10Has frameworks=+5, has details=+3, audit/logging category=+2
Category Weight×0.8–1.2High-risk categories (VULN, PRIVILEGE_ESCALATION…)=1.2×; medium=1.0×; low=0.8×

Severity floor

GCP's severity label is treated as authoritative. The computed score is never allowed to fall below the band that corresponds to GCP's severity:

GCP SeverityMinimum score enforced
CRITICAL75
HIGH55
MEDIUM / LOWNo floor

Priority assignment

Score rangePriority
75 – 100CRITICAL
55 – 74HIGH
35 – 54MEDIUM
0 – 34LOW

Stage 3 Compliance Detection

The detector (pkg/compliance/detector.go) parses two raw JSON fields from each finding:

  • finding.compliances → [{"standard":"CIS","version":"1.2","ids":["3.1"]}]
  • finding.compliance_details.frameworks → [{"name":"ISO27001","controls":["A.12.4"]}]

Framework names are normalized to canonical IDs:

Observed namesCanonical ID
CIS, CIS_BENCHMARK, CISGCP, CIS_GCPCIS
PCI, PCIDSS, PCI_DSSPCI_DSS
HIPAAHIPAA
SOC2, SOC 2, SOC_2SOC_2
ISO27001, ISO 27001, ISO_27001ISO_27001
NIST, NIST_CSF, NIST CSFNIST
GDPRGDPR

Results are stored on Finding.Violations and aggregated into Report.ComplianceSummary grouped by framework → control with occurrence counts.

Activated with --include-remediation. Populates Finding.Remediation for every finding (pkg/remediation/guidance.go).

Stage 5 AI Enrichment (optional)

Activated with --ai-enhance. Requires ANTHROPIC_API_KEY in the environment. Processes only CRITICAL findings, up to 4 concurrently (pkg/llm/enricher.go).

ParameterValue
Endpointhttps://api.anthropic.com/v1/messages
Modelclaude-haiku-4-5-20251001
Max tokens1024
HTTP timeout30 seconds
Concurrency4 parallel calls

The prompt includes: category, severity, resource name, description, CVE ID, CVSS score, and GCP's next_steps. The API returns JSON with three fields:

{
  "risk_rationale":     "2-3 sentence explanation of the security risk",
  "remediation_script": "complete runnable bash or python3 script",
  "remediation_lang":   "bash"
}

The AI output overwrites Finding.RiskScore.Rationale and Finding.Remediation.RemediationScript. If the API call fails for any reason, the original template-based content is preserved and a warning is logged — processing continues.

Stage 6 Filtering

Applied after all enrichment stages so that every finding has been fully scored before being reduced (cmd/analyzer/analyze.go → applyFilters). Filters are cumulative:

FlagBehavior
-p CRITICAL,HIGHKeep only findings with matching priority
-c FIREWALL,IAMKeep only findings with matching category (case-insensitive)
--project my-projectKeep only findings where ProjectID or ProjectDisplayName matches
--min-risk-score 70Keep only findings with RiskScore.Total ≥ 70
--max-risk-score 90Keep only findings with RiskScore.Total ≤ 90

Stage 7 Report Generation

Report object assembly (pkg/report/builder.go)

Before writing files, the Builder aggregates findings into a *models.Report with: priority counts, risk score statistics (mean, median, min, max, stddev), top categories/projects by finding count, per-category and per-project breakdowns, and a compliance summary.

Output formats

json.jsonComplete Report struct serialized as pretty-printed JSON

End-to-End Examples

Example 1 — Basic CSV analysis (default output)

GCP-Sec analyze findings.csv
  1. root.go dispatches to runAnalyze
  2. ExtractPositional identifies findings.csv as the positional argument
  3. parser.ParseFile opens the file, reads header, maps columns, parses rows → []*Finding
  4. runPipeline executes:
    • scoring.Engine.ScoreAll() — assigns RiskScore + Priority (8 goroutines)
    • compliance.Detector.DetectViolations() — parses JSON compliance fields
    • No remediation, no AI, no filters
    • report.Builder.Build() — aggregates into *Report
    • Writes findings-report.md and findings-report.html

Output: findings-report.md, findings-report.html

Example 2 — Full pipeline with AI enrichment

GCP-Sec analyze findings.csv \
  --include-remediation \
  --include-compliance \
  --ai-enhance \
  -o security-report.md
  1. Parse findings.csv → []*Finding
  2. Score all findings → risk scores + priorities
  3. Detect compliance violations → annotate findings
  4. Generate remediation guidance for every finding
  5. Generate automation scripts for CRITICAL findings
  6. Call Claude API (up to 4 concurrent) for each CRITICAL finding → overwrite rationale + script
  7. No filters applied
  8. Build *Report with compliance summary
  9. Write security-report.md + security-report.html
  10. Write remediation-scripts/*.sh and *.py

Example 3 — Live fetch with priority filter

gcloud auth application-default login

GCP-Sec fetch \
  --org-id 123456789012 \
  --days 14 \
  --priority critical,high \
  --include-remediation \
  -o scc-weekly.md
  1. Connect to GCP SCC API using Application Default Credentials
  2. List ACTIVE findings from the last 14 days (paginated, 1,000 per page)
  3. Convert protobuf findings → []*Finding
  4. Score, detect compliance, generate remediation + scripts
  5. Apply filter: keep only CRITICAL and HIGH priority
  6. Build report and write scc-weekly.md + scc-weekly.html
  7. Write remediation scripts for CRITICAL findings

Key Design Decisions

Modular, toggleable pipeline

Each stage (scoring, compliance, remediation, AI, filtering, reporting) is independently activated via flags. The shared runPipeline() function orchestrates them so that both analyze and fetch commands benefit from the same logic with zero duplication.

Severity floor preserves GCP's judgment

The computed multi-factor score is never allowed to downgrade a finding below the band corresponding to GCP's authoritative severity label. This prevents the scoring engine from contradicting GCP's expert assessment while still differentiating within each severity band.

Default output: two formats

With no format flags, the tool always generates both Markdown and HTML. This avoids the common frustration of getting only one format and having to re-run. The format default is an empty string, so any explicit --format value is treated as an override.

Graceful degradation

  • Missing CVSS / CVE / compliance data → scoring continues with those components at 0
  • AI API failures → original template-based remediation preserved; processing continues
  • CSV parse errors → logged as warnings; valid rows still processed

Concurrency

  • Scoring: 8 worker goroutines for high-throughput parallel scoring
  • AI enrichment: 4 concurrent Claude API calls to minimize latency

Single-file, self-contained HTML

The HTML report embeds all CSS and JavaScript inline so it can be opened in any browser without a server or additional assets. Print CSS is included so the report prints cleanly.

Column aliasing in CSV parser

GCP's CSV export format uses full API dotted paths (finding.category, resource.project_id). The parser's alias map handles both this format and simplified column names, making the tool work with both Console exports and gcloud CLI exports without user configuration.

Category-driven remediation

Remediation scripts, effort estimates, automation hints, and documentation links are all driven by the finding category rather than generic templates. This gives security teams targeted, actionable guidance for each specific vulnerability type.
