Section 1

What the tool does — the big picture

Google Cloud's built-in security service (Security Command Center, or SCC) continuously monitors your GCP projects and flags anything that looks wrong: open firewall ports, exposed storage buckets, unpatched containers, leaked credentials, and more. It can surface thousands of these findings at once — but it doesn't tell you which ones to fix first, or how.

The GCP Security Analyzer fills that gap. It ingests those raw findings — either pulled live from the GCP API or from a CSV export — runs them through a multi-factor risk-scoring algorithm, maps them against compliance frameworks, generates remediation scripts, and writes a clean, prioritized report in your choice of format (Markdown, HTML, JSON, or CSV).

📥 Ingest
Pull findings live from the GCP API or load a CSV export from Security Command Center.
🎯 Score & Prioritize
Every finding gets a risk score (0–100) based on severity, CVSS, exploitability, exposure, and compliance impact.
📄 Report
Outputs a prioritized, actionable report with per-finding remediation steps and optional AI-enhanced guidance.

Section 2

The Go programs explained

The codebase is split into three layers: a command layer (what you type at the terminal), a package layer (the domain logic), and an internal layer (shared utilities and data models). Here is what each file does.

Command Layer — what you run from the terminal

main.go
Entry Point main.go
The single entry point for the whole program. Its only job is to hand control to the command dispatcher and exit with the right status code (0 = success, 1 = error). Everything else happens elsewhere.
cmd/analyzer/root.go
Dispatcher root.go — CLI router
Reads the first word you type after the binary name and routes to the right command: analyze, fetch, stats, or filter. Also sets up the global logger (verbose / debug mode) before any command runs.
cmd/analyzer/analyze.go
Command analyze — process a CSV file
The most commonly used command. You give it a CSV file exported from GCP Security Command Center; it parses it, runs the full analysis pipeline, and writes the report. Supports filtering by priority, category, project, or risk score range, and can write separate report files per severity level (Critical-only, High-only, etc.).

Example: GCP-Sec analyze findings.csv --priority critical,high --include-remediation
cmd/analyzer/fetch.go
Command fetch — pull live from GCP
Instead of using a CSV file, fetch connects directly to the GCP Security Command Center API in real time. You supply your GCP Organization ID and a lookback window (e.g. the last 7 days), and it retrieves all currently active findings automatically. From that point on, the analysis pipeline is identical to the analyze command. You can optionally save the raw fetched data as a CSV file for later use.

Example: GCP-Sec fetch --org-id 123456789 --days 14 --save-csv raw.csv
cmd/analyzer/stats.go
Command stats — quick summary
A lightweight read-only command. It parses and scores a CSV file but does not write any report file. Instead it prints a quick dashboard to the terminal: total findings, breakdown by priority, risk score statistics (mean, median, standard deviation), top 10 categories, compliance framework violations, and the most-affected projects. Good for a fast situational overview before deciding what to focus on.
cmd/analyzer/filter.go
Command filter — slice and export
A utility command for data preparation. It reads a CSV, applies filters (priority, category, project, risk score range), and writes just the matching rows back out as a new CSV — useful for handing a focused subset to another team or loading into a spreadsheet.
cmd/analyzer/pipeline.go
Shared Logic pipeline.go — the analysis engine
The most important file in the command layer. Both analyze and fetch end up here. It runs every finding through the full processing chain in the correct order: scoring → compliance detection → remediation guidance → optional AI enrichment → filtering → report building → output writing. Separating this into its own file means the pipeline is identical regardless of where the findings came from.

Package Layer — the domain logic

pkg/fetcher/fetcher.go
Package fetcher
Connects to the GCP Security Command Center gRPC API, builds a time-filtered query for ACTIVE findings, and iterates through all paginated results (up to 1 000 per page). Returns a clean list of internal Finding objects.
pkg/fetcher/convert.go
Package fetcher / converter
Translates the raw GCP API response (a protobuf object) into the internal data model the rest of the program understands. Extracts CVE data, CVSS scores, compliance mappings, resource names, and timestamps.
pkg/parser/csv.go
Package parser
Reads a CSV file exported from GCP SCC and maps each row to an internal Finding struct. Handles malformed rows gracefully — bad rows are counted and skipped rather than crashing the program. Also parses embedded JSON fields (CVSS data, compliance details, etc.).
pkg/scoring/risk.go
Package scoring
The risk-scoring engine. Runs up to 8 goroutines in parallel to score all findings simultaneously. Each finding receives a composite score from six components (see Section 4). The engine also enforces severity floor values to prevent a high-severity GCP finding from ever scoring lower than the band GCP itself assigned.
pkg/compliance/detector.go & frameworks.go
Package compliance
Maps each finding to the compliance frameworks it violates: CIS Benchmarks, PCI-DSS, HIPAA, SOC 2, ISO 27001, NIST, and GDPR. The framework list is defined in frameworks.go with alias normalization, and the detector in detector.go applies the matching logic to each finding.
pkg/remediation/guidance.go & scripts.go
Package remediation
Generates human-readable remediation steps, estimated fix effort, automation potential, and documentation links for every finding. For CRITICAL findings it also generates a complete shell script with real resource names baked in, ready to run.
pkg/llm/enricher.go
Package llm (optional)
When the --ai-enhance flag is set and an Anthropic API key is present, this package sends each CRITICAL finding to the Claude AI API. Claude returns an improved risk explanation and a more detailed remediation script tailored to the specific resource and project. If the API key is absent the program runs identically without any degradation.
pkg/report/builder.go, html.go, markdown.go, json.go, csv.go
Package report
Assembles findings into a Report object and renders it in one or more output formats. The builder computes aggregate statistics (counts, risk distribution, top categories, project breakdown). Each renderer (HTML, Markdown, JSON, CSV) is a separate file so they can evolve independently.

Internal Layer — shared foundations

internal/models/finding.go, report.go, risk_score.go
Models Data structures
Defines the core data shapes shared by every package: Finding (a single security issue with all its attributes), RiskScore (the scoring breakdown), RemediationStep (guidance + scripts), and Report (the final assembled output).
internal/utils/logger.go, helpers.go, flags.go
Utils Shared helpers
Provides a structured logger with Info/Debug/Warn levels, math helpers (rounding, clamping, safe percentages), string utilities, and a flag-parsing helper that correctly separates positional arguments from named flags regardless of their order on the command line.

Section 3

How live findings are pulled from GCP

When you run the fetch command, the tool speaks directly to Google's Security Command Center API over an encrypted gRPC connection. Here is exactly what happens under the hood, in plain terms.

1
Authentication The tool uses your existing Google Cloud credentials — the same ones set up by running gcloud auth application-default login. No passwords or API keys need to be embedded in the tool itself.
2
Building the query The tool constructs a filter that asks for all findings where: (a) the status is ACTIVE — meaning GCP has not yet marked them as resolved, and (b) the event time is within your chosen lookback window (default: last 7 days). It targets your entire GCP organization at once, so findings from all projects under that organization are included.
3
Paginated retrieval GCP returns results in pages of up to 1 000 findings at a time. The tool automatically fetches every page until there are no more results, so you always get the complete picture even for large organizations. Progress is logged every 500 findings.
4
Data conversion Each raw GCP finding (a protobuf object from the API) is translated into the internal Finding format the rest of the program works with. This step extracts CVE identifiers, CVSS scores, zero-day and exploit-in-the-wild flags, public IP exposure, compliance framework mappings, and resource/project metadata from the nested API response.
5
Hand-off to the analysis pipeline At this point the findings look identical to what the analyze command would have produced from a CSV file. They flow into the same scoring, compliance, remediation, and report-generation steps described in the next section. There is no difference in the output quality between the live API mode and the CSV mode.
What authentication is needed? The tool requires a GCP service account (or your personal credentials) with the Security Center Findings Viewer role (roles/securitycenter.findingsViewer) on the organization. For CI/CD pipelines, a dedicated service account with this single role is recommended.

Section 4

Tutorial: pull & save live data as CSV

This walkthrough takes you from a fresh machine to a saved CSV file containing your organization's most recent active findings — and an analyzed report. Follow the steps in order. Each step builds on the previous one.

Step 1 — Install the tool

Clone the repository and build the binary. You need Go 1.21 or later.

# Clone and build
git clone https://github.com/wanaware/GCP-Sec.git
cd GCP-Sec
go build -o GCP-Sec .

# Verify it works
./GCP-Sec version

The binary is self-contained — no runtime dependencies, no installation wizard. Move it anywhere on your $PATH if you like.

Step 2 — Authenticate to GCP

Option A  Personal credentials (local use)

If you already have the gcloud CLI installed and are logged in to your GCP account, run:

gcloud auth application-default login

This writes a credentials file to your home directory. The tool picks it up automatically — no extra configuration needed.

Option B  Service account key (CI/CD or shared use)

If you have a JSON key file for a service account that has the Security Center Findings Viewer role (roles/securitycenter.findingsViewer), point the tool at it with an environment variable:

export GOOGLE_APPLICATION_CREDENTIALS=/path/to/service-account-key.json

Set this once per shell session (or add it to your shell profile). Never commit the key file to version control.

Step 3 — Find your GCP Organization ID

The fetch command requires your GCP Organization ID (a numeric ID, not the display name). Find it with:

gcloud organizations list

The output shows a table with DISPLAY_NAME, ID, and DIRECTORY_CUSTOMER_ID. Copy the value from the ID column — it looks like 123456789012. You will use this as the --org-id argument in the next steps.

Step 4 — Fetch live findings and save as CSV

This is the core command. It connects to the GCP Security Command Center API, downloads all active findings within your chosen lookback window, and saves the raw data as a CSV file you can open in a spreadsheet or feed back into the tool later.

Most common  Last 7 days, save CSV + generate report
# Replace 123456789012 with your real Organization ID
./GCP-Sec fetch \
  --org-id 123456789012 \
  --days 7 \
  --save-csv findings-$(date +%Y%m%d).csv

This saves a timestamped CSV (e.g. findings-20260309.csv) and immediately runs the analysis pipeline on the same data, writing scc-org-123456789012-report.md and .html alongside it.

Wider window  Last 30 days, Critical & High only
./GCP-Sec fetch \
  --org-id 123456789012 \
  --days 30 \
  --priority critical,high \
  --save-csv critical-high-$(date +%Y%m%d).csv \
  -o reports/critical-high-report.md

The --priority filter applies after the full dataset is fetched and scored, so the CSV contains only the matching findings while the report directory is set by the stem of the -o path.

Full options  AI-enhanced, all formats, verbose logging
# Set your Claude API key for AI-enhanced remediation on CRITICAL findings
export ANTHROPIC_API_KEY=sk-ant-...

./GCP-Sec fetch \
  --org-id 123456789012 \
  --days 7 \
  --save-csv raw-$(date +%Y%m%d).csv \
  --formats markdown,html,json \
  -d ./reports \
  --include-remediation \
  --include-compliance \
  --ai-enhance \
  --verbose

Writes three report files into ./reports/ (Markdown, HTML, JSON), saves the raw CSV, annotates compliance framework violations, and sends CRITICAL findings to the Claude API for AI-enhanced risk rationale and remediation scripts.

CSV only  Save raw data without generating a report

There is no "CSV only" mode — the tool always runs the analysis pipeline. To get just the CSV with minimal overhead, pass --format csv so the report itself is also written as CSV, then use only the --save-csv file in subsequent steps: 

./GCP-Sec fetch \
  --org-id 123456789012 \
  --days 7 \
  --save-csv raw-findings.csv \
  --format csv \
  -o scored-findings.csv

raw-findings.csv — the unsorted raw data exactly as fetched from GCP.
scored-findings.csv — the same data after scoring, filtering, and sorting by risk score.

Step 5 — Re-analyze the saved CSV (optional)

Once you have a CSV file you can run the analyze command on it as many times as you like — with different filters, formats, or options — without needing to hit the GCP API again. 

# Generate a full HTML + Markdown report from the saved CSV
./GCP-Sec analyze raw-findings.csv \
  --include-remediation \
  -o reports/full-report.md

# Filter down to just Critical, split into separate files per priority
./GCP-Sec analyze raw-findings.csv \
  --priority critical \
  --split-by-priority \
  -d reports/split/

# Quick stats summary — no report file written
./GCP-Sec stats raw-findings.csv

Quick reference — fetch command flags

Flag Required? Description
--org-id Required Your GCP Organization ID (numeric). Found with gcloud organizations list.
--days Optional Lookback window in days. Default: 7. Use 1 for the last 24 hours, 30 for the last month.
--save-csv Optional Path to save the raw fetched findings as CSV before analysis (e.g. raw-20260309.csv).
-o / --output Optional Output file path. The file stem becomes the base name for all generated report files.
-d / --output-dir Optional Write all report files into this directory instead of the current working directory.
--formats Optional Comma-separated output formats: markdown,html,json,csv. Default: markdown + html.
--priority Optional Filter results: critical, high, medium, low (comma-separated).
--include-remediation Optional Add step-by-step remediation guidance and shell scripts to the report. On by default.
--include-compliance Optional Map each finding to violated compliance controls (CIS, PCI-DSS, HIPAA, SOC 2, NIST, GDPR).
--ai-enhance Optional Use Claude AI to improve rationale and remediation scripts for CRITICAL findings. Requires ANTHROPIC_API_KEY.
--verbose / --debug Optional Show detailed progress logs during fetch and analysis.

What you get after running the command

📋 The raw CSV (--save-csv)
The findings exactly as returned by GCP — one row per finding, with all raw fields preserved. Use this to archive a point-in-time snapshot, share with another team, load into a SIEM, or re-analyze later with different options.
📄 The analyzed report
A prioritized, scored report in your chosen format(s). Findings are sorted by risk score (highest first), annotated with compliance violations and remediation steps, and summarized in an executive statistics block at the top.
🛠 Remediation scripts
For CRITICAL findings, a remediation-scripts/ subdirectory is created containing runnable shell scripts — one per finding — with the real resource names and project IDs already filled in.
📊 Terminal summary
A quick dashboard is always printed to the terminal: total findings, Critical / High / Medium / Low counts, mean risk score, and score range — so you get the headline numbers without opening any file.
Tip: run stats first. Before generating a full report on a large dataset, run ./GCP-Sec stats raw-findings.csv to see a breakdown by priority and category. This helps you decide which filters to apply when generating the report, avoiding an overwhelming multi-thousand-finding output.

Section 5

The analysis pipeline, step by step

Once findings are loaded — whether from the API or a CSV file — they go through the same seven-stage pipeline. Each stage enriches the findings with additional context.

Stage 1 — Risk Scoring (0–100 scale)

Every finding gets a numeric risk score assembled from six components. The final score is capped at 100, and a severity floor is enforced so that a GCP CRITICAL finding always scores at least 75, and HIGH at least 55 — even if vulnerability metadata is missing.

Base Severityup to 40 pts
Critical=40, High=30, Medium=20, Low=10 — GCP's own severity rating.
CVSS Scoreup to 30 pts
The industry-standard vulnerability severity score (0–10), scaled up.
Exploitabilityup to 20 pts
Bonus points for: actively exploited in the wild (+10), zero-day (+8), active exploit code available (+6), known CVE (+2).
Finding Classup to 10 pts
THREAT=10, VULNERABILITY=7, MISCONFIGURATION=5, OBSERVATION=2.
Internet Exposureup to 10 pts
Public IP address (+5), load-balancer facing (+3), data store (bucket/database) (+2).
Compliance Impactup to 10 pts
Bonus when the finding maps to a compliance framework (PCI-DSS, HIPAA, etc.).
After summing the six components the total is multiplied by a category weight (0.8×–1.2×) based on the type of finding — vulnerability and privilege-escalation categories get a 1.2× multiplier, and informational observations get 0.8×.
2
Compliance Detection The compliance detector checks each finding against seven frameworks — CIS Benchmarks, PCI-DSS, HIPAA, SOC 2, ISO 27001, NIST, and GDPR — and records which controls are violated. The framework names in GCP data come in many inconsistent formats; the detector normalizes them all to canonical identifiers before matching.
3
Remediation Guidance (optional, on by default) For each finding the remediation generator produces: a plain-English summary of what needs to change; step-by-step fix instructions; an estimated time to fix (Low/Medium/High); links to official GCP documentation; and an automation hint showing the relevant gcloud command. CRITICAL findings additionally receive a full, runnable shell script with the real resource names already filled in.
4
AI Enrichment (optional, requires --ai-enhance) If enabled, CRITICAL findings are sent to the Claude AI API. Claude reads the full finding context — category, severity, resource name, project, description, CVE — and returns an improved risk explanation and a more tailored remediation script. Up to four findings are enriched in parallel. The program works normally if this flag is absent.
5
Filtering The scored, enriched findings are filtered down based on flags you passed: priority level (Critical/High/Medium/Low), category keyword, specific GCP project, or a minimum/maximum risk score. Only the filtered subset appears in the final report.
6
Report Building The report builder assembles aggregate statistics — total counts by priority, risk score distribution (mean, median, std dev), top categories by frequency, and project-level breakdowns — then organizes findings by priority order (Critical first) ready for rendering.
7
Output Writing The report is rendered into one or more output formats — Markdown, HTML, JSON, CSV — and written to disk. Optionally, per-finding remediation shell scripts are written to a remediation-scripts/ subdirectory so engineers can review and run them directly.

Section 6

The SAST security layer

The GCP Security Analyzer is itself a security tool — so it holds itself to a high security standard. A SAST (Static Application Security Testing) pipeline watches the source code of the analyzer itself, scanning for secrets, code vulnerabilities, and known CVEs every time code is pushed to GitHub. Think of it as "security for the security tool."

Why does a CLI tool need SAST? Security tools often handle credentials, API keys, and sensitive data. If the tool itself has a hardcoded secret, a vulnerable dependency, or a code flaw, it becomes the weak link. SAST closes that gap automatically.

The four scanning tools

🔑 Gitleaks — Secret Scanner

Scans every file and git commit for hardcoded secrets: API keys, GCP service account private keys, auth tokens, and generic API key patterns.

  • Runs as a pre-commit hook locally (catches secrets before they are committed)
  • Also runs in CI on every push
  • Custom rules target GCP-specific credential formats
  • Blocks the build if a secret is found
🔬 Semgrep — Code Analysis

Reads the Go source code and looks for vulnerable patterns using both custom rules and the community Go/OWASP rulesets.

  • Hardcoded GCP credentials in code
  • Insecure TLS configuration (InsecureSkipVerify)
  • Command injection risks
  • Weak cryptography (MD5, SHA-1)
  • Sensitive variables being logged
  • Blocks the build on ERROR-level findings
📦 govulncheck — Dependency CVEs

Scans the Go module dependency tree against the official Go vulnerability database maintained by the Go security team.

  • Checks every imported library for known CVEs
  • Only flags vulnerabilities that are actually reachable in the code (smarter than just checking the dependency list)
  • Runs as an advisory — results are uploaded but do not automatically block merges
🐳 Trivy — Filesystem & Dep Scan

Scans the entire filesystem and compiled binary for HIGH and CRITICAL CVEs from multiple vulnerability databases (NVD, GitHub Advisories, OSV).

  • Broader database coverage than govulncheck alone
  • Detects OS-level vulnerabilities if a Dockerfile is present
  • Produces a SARIF report for the GitHub Security tab
  • Blocks the build on HIGH/CRITICAL findings

What happens after the scans

SARIF Upload
Gitleaks, Semgrep, and Trivy all produce reports in the SARIF format (a standard for security findings). These are automatically uploaded to the GitHub Security → Code Scanning tab, where findings are displayed with file locations, severity levels, and links to the relevant rule documentation.
GCS Storage
All four scan reports (JSON + SARIF) are uploaded to a Google Cloud Storage bucket (wanaware-sast-reports) keyed by commit SHA. Reports are retained for 365 days and are not publicly accessible. This creates a permanent, auditable record of every security scan result for every build.
Email Notification
After every scan run (pass or fail), an email notification is sent to the security team via SendGrid. The email includes the overall pass/fail status, the result of each individual scan tool, a link to the GitHub Security tab, and a link to the full reports in GCS. This job always runs even when a scan fails, so nothing is ever silently dropped.
Local Hooks
Developers can install a Git pre-commit hook (one make setup-hooks command) that runs Gitleaks before every commit. This catches secrets on the developer's machine before they ever reach GitHub, providing a "shift-left" safety net without waiting for CI.
